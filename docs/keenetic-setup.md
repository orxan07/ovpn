# Keenetic Hopper: настройка SSTP-клиента

Тестировано на **Keenetic Hopper KN-3811**, KeeneticOS 5.0.8.
Сервер — `infra/sstp/setup-sstp.sh` на VPS.

## 0. Предусловия

- На VPS поднят SSTP по инструкции [`infra/sstp/README.md`](../infra/sstp/README.md).
- Известны: `IP_СЕРВЕРА`, `ПОРТ`, `ЛОГИН`, `ПАРОЛЬ`.
- В админке Keenetic установлен компонент **«Клиент SSTP»**
  (System → Components → раздел «Применения» → **SSTP-клиент**, перезагрузка).

## 1. Создание подключения (Web UI)

**Интернет → Другие подключения → VPN-подключения → Добавить подключение**.

| Поле | Значение |
|------|----------|
| Имя подключения | `SSTP-VPS` |
| Тип (протокол) | **SSTP** |
| Подключаться через | `Провайдер` (основной WAN) |
| Адрес сервера | `IP_СЕРВЕРА:ПОРТ` — например `171.22.75.104:14942`<br>**Порт указывается через двоеточие в этом же поле**, отдельного поля нет |
| Имя пользователя | `keenetic` (или твой) |
| Пароль | из вывода `setup-sstp.sh` |
| Метод проверки подлинности | **MS-CHAP v2** (auto тоже сработает) |
| Размер MTU | `1400` |
| Подстройка TCP MSS | **Включить** (Auto / PMTU) |
| Не использовать для выхода в интернет | **Выкл** (т.е. галка снята → "Использовать для выхода в интернет" = ВКЛ) |
| Получать маршруты от удалённой стороны | **Вкл** |
| Использовать для доступа в интернет | **Вкл** |
| Проверять сертификат сервера | **Выкл** (cert self-signed) |

Сохранить → подключение должно показать «Подключено» в течение нескольких секунд.

> **Если «Не подключается»**: первая причина — порт указан **не** в поле адреса.
> Keenetic не даёт отдельного поля для SSTP-порта, дефолт = 443.
> Нужно прописать `IP:PORT` в одной строке.

## 2. Маршрутизация: пустить нужные устройства через VPN

Подход — **профили доступа в интернет** (бывший «Приоритеты подключений»).

**Сетевые правила → Приоритеты подключений → Политики**:

1. Создать политику, например `Policy0` (или использовать существующую `KWG`).
2. Список интерфейсов **сверху вниз** = приоритет:
   ```
   1. SSTP-VPS                ← основной выход
   2. Провайдер (ISP)         ← резерв
   ```
3. Применить политику к нужным устройствам:
   - **Сетевые правила → Приоритеты подключений → Устройства**
   - Найти устройство (например `iPhone 2` по MAC `xx:xx:xx:xx:xx:xx`)
   - Назначить ему политику `Policy0`.

Готово. Это устройство теперь ходит через VPS.

## 3. Проверка

На устройстве, привязанном к политике, открыть https://ifconfig.me/ip
— должен показать `IP_СЕРВЕРА`.

DNS-leak можно проверить на https://www.dnsleaktest.com/.

## 4. Полезные CLI-команды Keenetic

Зайти можно по SSH (`admin@<IP_роутера>`) или через **Системные настройки →
Диагностика → Командная строка**.

```bash
# просмотр статуса всех интерфейсов
show interface

# конкретно SSTP
show interface SSTP0           # имя зависит от того, какой по счёту

# счётчики (видно, идёт ли реально трафик)
show interface stat SSTP0

# таблица маршрутов
show ip route

# текущий running-config
show running-config

# ping/traceroute (если в Web UI «Сетевая диагностика»)
```

## 5. Чек-лист, если «подключено, но интернета нет»

1. **`security-level private`** на интерфейсе VPN.  
   В running-config у блока `interface SSTPx` должно быть
   `security-level private`. Иначе firewall режет inbound. Меняется так:
   ```
   interface SSTPx
     security-level private
   exit
   system configuration save
   ```

2. **`ip nat`** на интерфейсе VPN отсутствует.  
   В большинстве случаев `ip nat` ставится только на WAN, **не** на туннель.
   Проверь, что у тебя нет лишнего `ip nat SSTPx`. (Для туннеля от VPS он
   обычно не нужен — NAT для пользовательского трафика делает уже сам VPS.)

3. **«Использовать для выхода в интернет» (`ip global`)** на интерфейсе VPN.  
   Без этой галки интерфейс **не появляется** в списке выбора в политиках.
   Включается там же в свойствах подключения.

4. **MTU**.  
   1400 — хороший дефолт. Если страницы открываются медленно/частично, попробуй
   1380 → 1360.

5. **Pin маршрут до сервера** через WAN, чтобы не было петли.  
   Keenetic делает это автоматически (`ip route IP_СЕРВЕРА SSTPx auto` —
   **этого не должно быть**, должно быть наоборот через ISP). Если видишь
   маршрут к серверу через сам туннель — это и есть петля.

6. **Сертификат сервера** — выключи проверку, у нас self-signed.

## 6. Ограничения Keenetic Hopper, которые мы поймали по пути

- В стоковом образе **нет AmneziaWG**. WireGuard есть только классический.
  Для DPI-стойкости остаётся либо SSTP, либо ставить альтернативную прошивку.
- В CLI у `ping` нет флага `source`. Диагностика делается через **Web UI →
  Системные настройки → Диагностика**.
- Внутренний `ping` из CLI не всегда показывает реальную картину — лучше
  тестировать с реального клиента (Mac/iPhone), привязанного к политике.
