# DPI bypass: история выбора протокола

Эта заметка фиксирует, **что мы пробовали и почему пришли к SSTP**.
Контекст: домашний интернет МТС GPON, роутер Keenetic Hopper KN-3811
(KeeneticOS 5.0.8), VPS — Ubuntu 24.04 (`171.22.75.104`).

## Симптомы блокировки

На всех вариантах ниже handshake / TCP-сессия устанавливались успешно
(значит до VPS пакеты доходили), но **полезные данные после установления
сессии не проходили** — это классическая картина DPI, который пропускает
control-plane и режет data-plane.

## Что пробовали

| # | Протокол | Порт | Результат | Почему не подошло |
|---|----------|------|-----------|-------------------|
| 1 | WireGuard | UDP/443  | handshake есть, односторонний трафик | DPI распознаёт WG handshake и режет последующие пакеты |
| 2 | WireGuard | UDP/51820 | то же | смена порта не помогает — distinguisher по содержимому |
| 3 | WireGuard | UDP/4500 | то же + потери | UDP/4500 у МТС вообще видит как IPSec NAT-T → блокирует |
| 4 | OpenVPN | TCP/14942 | TCP-сессия есть, 100% потерь после `tls-crypt` | TCP-сигнатура OpenVPN всё ещё распознаётся |
| 5 | OpenVPN | UDP/4500 | то же — `tcpdump` показал ESP, всё блок | UDP/4500 = IPSec в глазах DPI |
| 6 | **SSTP**  | TCP/14942 | **работает стабильно, RX/TX растут** | TLS поверх TCP неотличим от HTTPS |

## Почему именно SSTP

- **TLS-обёртка**: всё после первых байт — обычный TLS-трафик к произвольному
  TCP-порту. DPI не может отличить от обычного браузерного HTTPS без
  поведенческого анализа сессии.
- **Нативная поддержка в Keenetic**: «Интернет → Другие подключения → VPN →
  тип SSTP». Не нужен сторонний пакет (как у `awg` для WireGuard, которого
  на Hopper нет).
- **Стабильность**: TCP-перенос вместо UDP даёт меньше «магии» с MTU/PMTU и
  отсутствие проблем с UDP-флоу-control операторов.

## Минусы и компенсации

| Минус | Компенсация |
|-------|-------------|
| TCP-over-TCP (если поверх SSTP бегает другой TCP) — может приводить к meltdown при потерях | На стабильном линке (наш случай) проблема не проявляется |
| MS-CHAP v2 — старая аутентификация | Делаем длинный случайный пароль, не используем словарные |
| Self-signed cert | На Keenetic выключаем «проверку сертификата». На Windows можно ставить cert в Trusted Root |
| Нет нативных клиентов на macOS/iOS | Цель проекта — пустить **роутер целиком**, клиенты ходят через Wi-Fi домашней сети |

## Что дальше можно попробовать (если SSTP начнут блокировать)

1. **Переезд SSTP на 443/TCP** + реальный домен + Let's Encrypt cert — ещё ближе к «обычному HTTPS». (Сейчас 443 у нас занят nginx — нужно либо разнести по доменам/SNI, либо вынести админку на другой порт.)
2. **WireGuard с обфускацией (`amneziawg`)** — но на Hopper его нет, нужен MikroTik / OpenWrt / Linux router.
3. **VLESS + Reality** через sing-box — добавляется к существующей админке.
4. **SSTP + домашний DPI-evasion**: добавить fake SNI / domain fronting на стороне VPS (nginx с SNI-роутингом 443 → accel-ppp).
