# DPI bypass: история выбора протокола

Эта заметка фиксирует, **что мы пробовали и почему пришли к SSTP**.
Контекст: домашний интернет МТС GPON, роутер Keenetic Hopper KN-3811
(KeeneticOS 5.0.8), VPS — Ubuntu 24.04 (`171.22.75.104`).

> ⚠️ Важная оговорка: всё ниже — это эмпирика **в одной конкретной связке**
> (Keenetic за домашним МТС). На других клиентах с того же VPS WireGuard
> прекрасно работает (iPhone напрямую, Mac, мобильный интернет). Не делай
> из этого вывод «WG/OpenVPN везде блокируются» — у нас этого факта нет.

## Симптомы

На WG/OpenVPN handshake / TCP-сессия устанавливались успешно (значит до VPS
пакеты доходили), но **полезные данные после установления сессии не проходили**.
RX>0, TX=0 (или наоборот). Это похоже на работу DPI, который пропускает
control-plane и режет data-plane, но **точную причину мы не диагностировали
до конца** (см. ниже «Что мы НЕ проверили»).

## Что пробовали

| # | Протокол | Порт | Результат с Keenetic+МТС |
|---|----------|------|--------------------------|
| 1 | WireGuard | UDP/443  | handshake есть, односторонний трафик |
| 2 | WireGuard | UDP/51820 | то же |
| 3 | WireGuard | UDP/4500 | то же + бóльшие потери |
| 4 | OpenVPN | TCP/14942 | TCP-сессия есть, 100% потерь после `tls-crypt` |
| 5 | OpenVPN | UDP/4500 | в `tcpdump` ESP, ничего внутри не ходит |
| 6 | **SSTP**  | TCP/14942 | **работает стабильно, RX/TX растут** |

С iPhone напрямую (мобильный + домашний Wi-Fi через тот же Keenetic, но
на iPhone стоит native WG-клиент **в обход** туннеля Keenetic) — WG работает.
То есть проблема возникает **именно когда WG/OpenVPN-пакеты исходят из
роутера в WAN провайдера**.

## Что мы НЕ проверили (честно)

Любая из этих проверок дала бы окончательный ответ, но мы остановились
раньше — потому что нашли работающее решение, а тебе нужен был интернет.

- `tcpdump -ni any 'udp and host KEENETIC_WAN_IP'` на VPS во время
  активного WG-туннеля с роутера — увидеть, **долетают ли вообще TX-пакеты**
  от Keenetic. Это сразу разделило бы «провайдер режет на исходящем» vs
  «Keenetic не отправляет / NAT ломает».
- Подключить в Keenetic USB-модем другого оператора и повторить тест WG —
  если заработает, причина в МТС.
- WG на «доверенных» портах: UDP/53, UDP/123 (NTP), UDP/500.
- L2TP/IPsec.
- OpenVPN с `obfsproxy`/`Cloak`/`scramble` (Keenetic нативно не умеет, нужно
  поднимать обёртку на стороне VPS).
- AmneziaWG — на Hopper в маркете нет.

Возможные причины кроме DPI: CGNAT провайдера, особенности
WG/OpenVPN-стека Keenetic, MTU-проблемы на конкретном PPPoE/IPoE-аплинке.

## Сравнение SSTP vs WireGuard vs OpenVPN

### Производительность

| | WireGuard | OpenVPN | SSTP |
|---|---|---|---|
| Транспорт | UDP | UDP/TCP | **только TCP** |
| Шифрование | ChaCha20-Poly1305 (быстрое) | AES (с AES-NI быстро) | TLS+MPPE — ниже (CPU-bound на старых чипсетах роутера) |
| Overhead | минимальный (~32 байта/пакет) | средний (~50-60) | **высокий** (TLS+PPP+IP) |
| Latency на чистом линке | лучшая | средняя | средняя |
| TCP-over-TCP проблема | нет (UDP) | нет (UDP) | **есть** (если внутри SSTP бежит TCP, при потерях бывает meltdown) |
| Реальная скорость на Keenetic | ~до гигабита | ~300-500 Мбит/с | ~50-150 Мбит/с (упирается в TLS+PPP CPU) |

### Устойчивость к DPI

| | WireGuard | OpenVPN | SSTP |
|---|---|---|---|
| Сигнатура протокола | характерный handshake (4 типа сообщений) | TLS-handshake с особенностями | **обычный TLS** |
| Можно перепутать с | ничем — WG узнаваем | ничем — OpenVPN-fingerprint известен | HTTPS-трафиком к любому серверу |
| Помогает ли смена порта | нет (DPI смотрит на содержимое) | нет | да (ставь на 443 — становится «как будто к веб-серверу») |
| Обход в нашей сетке | нужна обфускация (AmneziaWG / wstunnel) | нужна обфускация (obfsproxy / Cloak) | **работает as-is** |

### Удобство для роутера (Keenetic Hopper)

| | WireGuard | OpenVPN | SSTP |
|---|---|---|---|
| Нативная поддержка в KeeneticOS | да | да | **да** |
| Нужен сторонний компонент | нет | нет | нет |
| Конфиг через UI | да (импорт `.conf`) | да (импорт `.ovpn`) | да (Адрес / Логин / Пароль — 3 поля) |
| Можно дать пароль и логин — без файлов и QR | нет | нет (нужен `.ovpn`) | **да** |
| Создать клиента занимает | копирование файла | копирование `.ovpn` (~10 KB) | **2 строки в `chap-secrets`** |
| Self-signed cert требует | — | импорт CA | импорт CA на Windows; на Keenetic — отключить проверку |

### Безопасность

| | WireGuard | OpenVPN | SSTP |
|---|---|---|---|
| Криптография | современная, минималистичная | современная (TLS 1.3) | TLS-обёртка современная, **но внутри MS-CHAP v2** |
| Аутентификация | по ключам (асимметрия) | сертификаты или username/password+TLS | **MS-CHAP v2 (логин+пароль)** — слабый алгоритм, ломается оффлайн при перехвате handshake |
| Forward secrecy | да (Curve25519) | да | да (только для TLS-обёртки, MS-CHAP — нет) |
| Можно отозвать одного клиента | удалить peer | revoke cert | удалить строку из `chap-secrets` (удобнее всех) |
| Утечка пароля = | не применимо | пароль нужно отдельно сменить | **прямой доступ ко всему туннелю**, нужен длинный случайный пароль |

### Эксплуатация

| | WireGuard | OpenVPN | SSTP (accel-ppp) |
|---|---|---|---|
| Установка на VPS | `apt install wireguard` | `apt install openvpn` | **сборка из исходников** в Ubuntu 24.04 (в репозиториях нет) |
| Зависимости | минимальные | средние | `cmake`, `libssl-dev`, `liblua`, `libpcre`, build-tools |
| Размер бинаря | <1 MB | ~1 MB | ~5 MB |
| Логи | `wg show` | `/var/log/openvpn/*.log` | `/var/log/accel-ppp/*` + `accel-cmd show sessions` |
| Управление сессиями в runtime | `wg-quick down/up` | через management socket | `accel-cmd disconnect <ifname>` (удобно) |
| Масштабируемость (>100 клиентов) | отлично | отлично | хорошо, но больше CPU на TLS |

## Когда что использовать (наш кейс)

- **iPhone напрямую (мобильный/Wi-Fi другой провайдер)** → **WireGuard**.
  Самое быстрое, минимум CPU на телефоне, отличное батарейное поведение.
- **iPhone на сетях с активным DPI (Мегафон ранее)** → **sing-box-app** на iPhone
  с обфускацией. Это уже не наш WG — это внешний клиент.
- **Mac / десктоп** → **WireGuard**. Та же причина.
- **Keenetic за домашним МТС (наш единственный «больной» кейс)** → **SSTP**.
  Только потому что WG/OpenVPN из этой связки не пошли. Если бы пошли —
  поставили бы WG (он быстрее, легче, безопаснее).
- **Windows-ноут гостя, к которому нужно дать VPN на 5 минут** → **SSTP**.
  Нативный клиент в системе, не нужно ничего ставить, дал логин+пароль и всё.

## Почему именно SSTP помог в нашем случае

Гипотеза (не доказана): TLS-обёртка SSTP неотличима от обычного HTTPS-трафика
к произвольному TCP-порту. Если домашний МТС-DPI работает по сигнатурам
content-plane WG/OpenVPN, то SSTP он распознать не может — это валидный TLS,
байты после первых 5 (TLS record header) — зашифрованы.

Возможно через год МТС обучит DPI отличать SSTP от обычного HTTPS по
поведенческим признакам (длина handshake, паттерны размеров пакетов). Тогда
придётся переезжать на VLESS+Reality / sing-box server-side. Но это уже
другая инженерная задача.

## Что дальше (если SSTP начнут блокировать)

1. **SSTP на 443/TCP с реальным доменом и LE-сертификатом** — ещё ближе к
   обычному HTTPS. Сейчас 443 у нас занят nginx (админка) — нужно либо
   разнести по доменам/SNI-роутингу, либо вынести админку.
2. **VLESS + Reality** через sing-box на стороне VPS. Маскируется под TLS
   к реальному домену (например `www.microsoft.com`). Требует sing-box-клиент
   на устройстве — на Keenetic его нет, поэтому только для прямых iPhone/Mac.
3. **WireGuard с обфускацией (`amneziawg`)** — нужен другой роутер
   (MikroTik / OpenWrt / mini-PC с Linux).
4. **Hysteria2 / TUIC** через sing-box — UDP-обфускация на современных
   протоколах. Та же проблема — нет на Keenetic.

## TL;DR

> SSTP в нашей задаче выбран не потому что он лучший по техническим параметрам
> — по большинству пунктов **он хуже WG**: медленнее, тяжелее, со старой
> аутентификацией, требует сборки из исходников и страдает от TCP-over-TCP.
> Выбран потому что это **единственный из трёх**, который сработал в связке
> Keenetic + МТС + наш VPS. Когда работают все три — выбираем WG.
